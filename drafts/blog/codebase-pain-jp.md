# ペイン駆動開発

## 発端

今日、GitHubのIssueをアサインされた。Laravelのイベントリスナーが2つ。ほぼ同一、それぞれ130行。同じオーケストレーションロジックが重複していた：エンティティのロード、複数の外部APIにまたがる同期ライフサイクルの管理、5つの`UseCase`クラスの順次呼び出し、エラーハンドリング。

## 「まあいい」修正案

Issueに提案されていた修正（悪くはないが、おそらく半分はAI生成）は、共通ロジックを`SyncUseCase`に抽出して、両方のListenerからそれを呼び出すというものだった。

動く修正だ。自分でやれば1時間。AIなら10分。

**でも、そうしなかった。**

## 違和感

脳の奥に「あのムズムズ」が来た。「おい、UseCaseがServiceを呼んで、そのServiceがまたUseCaseを呼ぶことになるぞ」という声。

まあ本番では動く。コードに感情はないから。

でも結果として、N階層にネストしたUseCaseが生まれる。booleanフラグが増え、matchケースだらけになる。そして相変わらず100行の命令型コードが残る。

問題はリスナーの重複そのものじゃなかった。**命名と層構造**だった。

その「ゴッドクラス」UseCase は多くのことをやりすぎていて、同じ層の「兄弟」クラスたちをオーケストレーションしていた。

## 本当の修正：足りなかった層

必要だったのは、その上の層だ。`Workflow`。「オーケストレータープロセス」と呼ぶ人もいる。名前はなんでもいい、「ただのUseCaseのひとつ」でなければ。

そのために、アプリケーション層にサブ層をひとつ追加した。

```php
$this->workflowPipeline
    ->send($payload)
    ->steps([$this->updateListing, $this->forceSync])
    ->skippable([$this->syncLeadTime], when: fn() => $payload->withLeadTimeSync)
    ->steps([$this->syncListing, $this->syncPromotion])
    ->run();
```

Listenerは130行から、こうなった：

```php
public function handle(SomeDomainEvent $event): void
{
    $this->flow->run($event->targetId);
}
```

## おまけ

そして**文字通りたくさんの**他のListener、ハンドラー、その他のコードが、今後この恩恵を受ける。

---

*「過設計だ！不要な複雑さを増やしてる！」と後ろから叫ぶ声が聞こえる気がする。*

---

## AIがやっていたら

あのIssueをAIエージェントに投げていたら、エージェントは正しいものを出していただろう。一見「過度に複雑ではない」ものを。

共通ロジックを`GigaChadSyncUseCase`に抽出して、Listenerを薄くして、レビューを通過させる。DRY、一貫性あり、完了。

動く。

でも間違っていた。「論理的に」間違いではない。「機能的に」間違いでもない。どちらかというと「構造的に」間違い。「将来性」の観点で間違い。

「Services」と「UseCases」が**完全な無秩序**の中でお互いを呼び合うコードが増えるだけ。意味のないコード。ただの命令の塊。長期的に見て、時間・エネルギー・メンタルバンドワイス、そして**トークン**を節約する「大局観」がない。

誰もそのシステムについて推論できなくなる。なぜならそのシステムは、推論されることを前提に設計されていないから。ユニットテストを通すため、AIコードレビューを通すため、GitHubのIssueをクローズするために設計され、そして**忘れられる**。

## 「AIネイティブ」のコスト

これが、**無批判に行われる**AIネイティブ開発の、真の、過小評価されたコストだと思う。

今日書くコードではない。正直、そのコードはたいてい良く、動き、人間が見落とすバグすら捕まえる。コストは**アーキテクチャ的洞察の喪失**だ。自分たちが何を作っているか、どう作るべきかを考えさせる「摩擦」の喪失だ。

1つの同期バリアントを追加するために6つのファイルを触らなければならない開発者、新しいステップを挿入する場所を探して130行のListenerをたどらなければならない開発者は、**腹が立つ**。そしてその腹立ちには価値がある。長期的にコードベースをより堅牢に保つ力がある。

*「アーキテクチャ的洞察なんて要らない、AIはどんなコードベースでも理解してコードを書ける」*と言う人には、こう問い返したい。「**トークンにいくら使う気がありますか？**」

そういった洞察は、将来のAIエージェントにとっても有益だと信じている：コンテキストが小さく、より論理的で、より宣言的で、構造化されているから。

直感に反するように見えるかもしれないが、アーキテクチャ的洞察を磨くことは、コードベースをコードの「[caveman](https://github.com/juliusbrussee/caveman)」にすることと同じだ。

## 違いは何か

結局、違いは**感情**だ。

開発者はタスクの痛みを感じる。エージェントはただ突き進む。6つのファイルを触り、チケットをクローズし、文句も言わず、摩擦もなく、パターン認識も働かない。

「メモリ」や「スキル」の話をいくらしても、エージェントはセッションをまたいでリセットされる。私たちが持っている「痛み→記憶→反応」のループがない。

> 「これ、こんなに難しくあるべきじゃない。」

少なくとも自分は、それがより良いコードを書く原動力になっている（あるいは、エージェントにそう指示する原動力に）。

## では、どうするか

はっきり言っておく：「エージェントを使うな」という話ではない。自分も毎日使っている。ただ、作業を投げて、テストとレビューを通るものが出てくるのを待って、コードをちょっと眺めて、先に進む——というやり方はしていない。

自分で試してみてほしい。疑問を持ってほしい。テストしてほしい。チケットの裏に何か隠れていないか考えてみてほしい。いつもあるとは言わない。バグ修正はバグ修正で、AIに60秒で解かせるべき場合もある。

でも時には、立ち止まって考えるべきだ。

結局のところ：1日単位では、他の開発者より少し遅いかもしれない。

でも長期的に積み重ねると、コードベースをより維持しやすく、より安く保っていると確信している。**次の開発者が人間であろうとAIであろうと関係なく。**
